# External Configuration Pattern

US-07 externalizes environment-specific values through Kubernetes ConfigMaps and Secrets generated by the `helm/circleguard` chart.

## Pattern

Stable service identity remains in each `application.yml`:

- `spring.application.name`
- `server.port`
- framework defaults and serializers

Environment-dependent values are resolved from environment variables. The Helm chart injects them with:

```yaml
envFrom:
  - configMapRef:
      name: <service>-config
  - secretRef:
      name: <service>-secret
```

Each ConfigMap also sets `SPRING_PROFILES_ACTIVE` from `global.springProfilesActive`, so `values-dev.yaml`, `values-stage.yaml`, and `values-prod.yaml` control the active Spring profile without changing the jar.

## Environment Overrides

Use environment-specific values files:

```bash
helm template circleguard ./helm/circleguard -n circleguard -f helm/circleguard/values-dev.yaml
helm template circleguard ./helm/circleguard -n circleguard -f helm/circleguard/values-stage.yaml
helm template circleguard ./helm/circleguard -n circleguard -f helm/circleguard/values-prod.yaml
```

Override image registry and secrets in CI/CD or a private values file:

```bash
helm upgrade --install circleguard ./helm/circleguard \
  -n circleguard \
  -f helm/circleguard/values-prod.yaml \
  --set global.imageRegistry=<acr-login-server> \
  --set services.auth-service.secret.JWT_SECRET=<rotated-secret>
```

## Secret Rotation

For development and stage:

```bash
kubectl edit secret circleguard-auth-service-secret -n circleguard
kubectl rollout restart deployment/circleguard-auth-service -n circleguard
kubectl rollout status deployment/circleguard-auth-service -n circleguard
```

For production, replace direct Secret editing with External Secrets Operator or a cloud secret manager-backed sync. The chart uses `stringData` intentionally so the migration path is simple and readable during Sprint 1.
